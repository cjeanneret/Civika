# Audit sécurité Civika - 2026-03-14

## Contexte et méthode

Audit statique et outillé du monorepo Civika (backend Go, frontend TypeScript/Next.js, configuration runtime), sans correction de code à ce stade.

Approche :
- Revue manuelle des surfaces d'entrée/sortie, des logs, des appels réseau, de la configuration et des dépendances.
- Exécution de vérifications non destructives.
- Association des constats à des standards reconnus.
- Priorisation par sévérité décroissante.

## Référentiels utilisés

- OWASP ASVS 5.0
- OWASP Top 10 (catégories 2021, toujours pertinentes pour la classification des risques web)
- OWASP REST Security Cheat Sheet
- OWASP Input Validation Cheat Sheet
- Go Security Best Practices ([go.dev/security](https://go.dev/doc/security/))
- `govulncheck` officiel Go ([documentation](https://pkg.go.dev/golang.org/x/vuln/cmd/govulncheck))
- TypeScript strict mode ([tsconfig strict](https://www.typescriptlang.org/tsconfig/strict.html))

## Périmètre audité

- Backend : `backend/cmd/civika-api`, `backend/internal/http`, `backend/internal/security`, `backend/internal/services`, `backend/internal/rag`, `backend/config`
- Frontend : `frontend/src`, `frontend/next.config.ts`, `frontend/tsconfig.json`, `frontend/package.json`
- Configuration/Ops : `README.md`, `.env.example`, `.env.test`, `docker-compose.yml`, `backend/go.mod`

## Vérifications exécutées

- Backend :
  - `go test ./...` -> OK
  - `go vet ./...` -> OK
  - `go run golang.org/x/vuln/cmd/govulncheck@latest ./...` -> 3 vulnérabilités affectant le code (stdlib)
- Frontend :
  - `npm run lint` -> OK
  - `npx tsc --noEmit` -> OK
  - `npm audit --omit=dev` -> 1 vulnérabilité critique (`next`)

Note : `pnpm` et `golangci-lint` ne sont pas disponibles sur l'environnement d'exécution de cet audit.

## Synthèse exécutive

- **Critique** : dépendance frontend `next@15.2.0` vulnérable selon `npm audit`.
- **Élevé** : 3 avis de sécurité CVE/GO sur la stdlib détectés par `govulncheck` dans des chemins de code exécutés.
- **Élevé** : absence de limitation de débit sur l'endpoint coûteux `POST /api/v1/qa/query` (risque de DoS applicatif).
- **Moyen** : validation incomplète de `X-Request-Id` (spoofing/injection dans les logs).
- **Moyen** : parsing JSON non strict en fin de flux (`decode` unique sans vérification EOF).
- **Moyen** : transport DB permissif (`POSTGRES_SSLMODE=disable` par défaut, également dans les exemples d'env).
- **Moyen** : URL de citation non validée avant rendu de lien externe.
- **Faible** : durcissement des headers/CSP absent dans l'app Next (si exposée directement sans proxy backend).

## Findings (triés par sévérité décroissante)

### CIV-SEC-001 - Critique - Composant frontend vulnérable (`next`)

- **Surface** : dépendances frontend
- **Preuve** :
  - `frontend/package.json` épingle `next: 15.2.0`.
  - `npm audit --omit=dev` remonte 1 vulnérabilité critique sur `next` avec plusieurs advisories.
- **Risque** : exposition à des failles documentées (incluant SSRF/DoS/exposition de code source selon advisories), avec impact potentiel direct sur la surface web.
- **Référence standard** :
  - OWASP Top 10 A06 (Vulnerable and Outdated Components)
  - OWASP ASVS (gestion des composants tiers)
- **Recommandation** : planifier une mise à niveau contrôlée de `next` vers une version corrigée, avec validation de régression sécurité/fonctionnelle.
- **Effort estimé** : moyen

### CIV-SEC-002 - Élevé - Vulnérabilités stdlib Go affectant des traces exécutées

- **Surface** : runtime backend Go
- **Preuve** :
  - `govulncheck` détecte :
    - GO-2026-4602 (`os`)
    - GO-2026-4601 (`net/url`)
    - GO-2026-4337 (`crypto/tls`)
  - Le scan indique des traces atteignables dans le code (`main.go`, `query.go`, `ingest.go`).
- **Risque** : exposition à des vulnérabilités connues dans la chaîne d'exécution HTTP/URL/TLS tant que le toolchain n'est pas corrigé.
- **Référence standard** :
  - Go Security Best Practices (mises à jour régulières + scan des vulnérabilités)
  - OWASP Top 10 A06
- **Recommandation** : mettre à jour la version Go (toolchain/stdlib) vers une version corrigée, puis relancer `govulncheck` en CI.
- **Effort estimé** : faible à moyen

### CIV-SEC-003 - Élevé - Endpoint QA coûteux sans rate limiting

- **Surface** : `POST /api/v1/qa/query`
- **Preuve** :
  - Route exposée dans `backend/internal/http/router.go`.
  - Aucun middleware de rate limiting/throttling détecté (`rg` sans résultat sur `rate|throttle|limiter`).
  - L'endpoint déclenche embeddings + recherche vectorielle + summarization (`backend/internal/services/qa_service.go`).
- **Risque** : déni de service applicatif (saturation CPU/IO/LLM) par envoi massif de requêtes.
- **Référence standard** :
  - OWASP Top 10 A04 (Insecure Design) / A08 (Integrity and misuse of pipeline)
  - OWASP REST Security Cheat Sheet (abuse prevention, API hardening)
- **Recommandation** : définir une stratégie de protection (rate limiting par IP/réseau/proxy, quotas, circuit breaker, budget de timeout).
- **Effort estimé** : moyen

### CIV-SEC-004 - Moyen - `X-Request-Id` non validé (spoofing/injection dans les logs)

- **Surface** : middleware HTTP
- **Preuve** :
  - `backend/internal/http/middleware.go` lit `X-Request-Id` côté client et le réutilise tel quel.
  - Cette valeur est ensuite loggée dans `accessLogMiddleware`.
- **Risque** : pollution des logs, corrélation trompeuse des événements, possibilité d'injection de caractères non attendus.
- **Référence standard** :
  - OWASP Top 10 A09 (Security Logging and Monitoring Failures)
  - OWASP ASVS (intégrité des journaux)
- **Recommandation** : valider format/longueur/charset ou régénérer systématiquement un ID côté serveur.
- **Effort estimé** : faible

### CIV-SEC-005 - Moyen - Parsing JSON non strict en fin de flux

- **Surface** : décodage du body de requête QA
- **Preuve** :
  - `backend/internal/http/handlers.go` utilise `decoder.Decode(&payload)` + `DisallowUnknownFields()`.
  - Pas de seconde lecture pour imposer EOF strict (acceptation possible de données trailing selon payload).
- **Risque** : ambiguïté d'interprétation de requête et surfaces de contournement de validation.
- **Référence standard** :
  - OWASP Input Validation Cheat Sheet (validation stricte au plus tôt)
  - OWASP ASVS (robust input handling)
- **Recommandation** : imposer une vérification de fin de flux JSON après le décodage principal.
- **Effort estimé** : faible

### CIV-SEC-006 - Moyen - Configuration DB permissive (`sslmode=disable`)

- **Surface** : configuration Postgres
- **Preuve** :
  - Défaut `POSTGRES_SSLMODE=disable` dans `backend/config/config.go`.
  - Valeur également présente dans `.env.example` et `.env.test`.
- **Risque** : si réutilisé hors environnement strictement local, risque d'interception/MITM des flux DB.
- **Référence standard** :
  - OWASP Top 10 A05 (Security Misconfiguration)
  - OWASP ASVS (secure transport)
- **Recommandation** : expliciter le mode local vs prod et exiger TLS DB hors développement.
- **Effort estimé** : faible

### CIV-SEC-007 - Moyen - Liens externes non validés avant rendu

- **Surface** : affichage des citations Q&A
- **Preuve** :
  - `frontend/src/components/qa/QAResponseStack.tsx` rend `href={citation.url}` directement.
  - `backend/internal/services/qa_service.go` transmet `SourceURI` en sortie (`Citation.URL`) sans allowlist explicite du schéma.
- **Risque** : redirection vers des liens malveillants/phishing si la source est compromise ou les données contaminées.
- **Référence standard** :
  - OWASP ASVS (output handling / URL validation)
  - OWASP Input Validation Cheat Sheet (allowlist)
- **Recommandation** : valider les schémas/hôtes autorisés avant exposition frontend (minimum `https`).
- **Effort estimé** : faible à moyen

### CIV-SEC-008 - Faible - Hardening frontend incomplet (headers/CSP non définis dans Next)

- **Surface** : app Next.js
- **Preuve** :
  - `frontend/next.config.ts` contient uniquement `reactStrictMode`.
  - Aucun `headers()` custom ni middleware frontend de sécurité détecté.
- **Risque** : si le frontend est exposé directement sans reverse proxy durci, politique de headers potentiellement insuffisante.
- **Référence standard** :
  - OWASP Top 10 A05
  - OWASP ASVS (secure headers policy)
- **Recommandation** : documenter le point d'application des headers (proxy ou Next) et ajouter un garde-fou explicite.
- **Effort estimé** : faible

## Points positifs déjà en place

- Limitation de la taille du body HTTP (`http.MaxBytesReader`) dans le middleware backend.
- Validation stricte de nombreux paramètres (`limit/offset/id/lang`) et `DisallowUnknownFields` pour le JSON.
- Headers de sécurité backend présents (`X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, CSP de base).
- `Content-Type` JSON explicite et erreurs API non verbeuses.
- Typage frontend strict (`strict: true`, `noImplicitAny: true`).
- Mode RAG explicite (`local` vs `llm`) avec erreurs de configuration explicites (pas de fallback silencieux).

## Limites de cet audit

- Pas de test DAST/fuzz/replay en environnement de production.
- Pas de scan image/container (ex. : Trivy) dans cette passe.
- Pas de revue d'infrastructure cloud/reverse proxy/WAF.
- Analyse concentrée sur le code et les configurations du dépôt à date.

## Plan de remédiation proposé (sans patch dans cette phase)

1. Traiter en priorité les composants vulnérables :
   - upgrade `next` vers une version corrigée,
   - upgrade du toolchain Go corrigé, puis rerun `govulncheck`.
2. Réduire le risque de DoS sur `/api/v1/qa/query` :
   - rate limiting + quotas + garde-fous de charge.
3. Renforcer la robustesse des entrées :
   - validation de `X-Request-Id`,
   - décodage JSON strict EOF.
4. Durcir les configurations de transport et de sortie :
   - politique TLS DB explicite hors local,
   - validation stricte des URLs de citations,
   - clarification/standardisation de la politique de headers côté frontend.
